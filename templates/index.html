What is Workflow Buddy?

Workflow Buddy fills the gaps we've seen in Workflow Builder functionality. This includes:

  • an HTTP Request/Outgoing Webhook Step
  • Debug Mode while working on your workflows
  • failure notifications if a Buddy Step fails
  • .... and much more - refer back to the docs if you are interested in more info.

Beta?

You're experiencing the Beta of the hosted/cloud version of Workflow Buddy. The early self-hosted versions of Buddy have been running for non-critical functionality at multiple companies over the past ~6 months, and we are now piloting a hosted Beta. We are considering charging in the future to help defray support costs - so if that is a blocker, you should consider the self-hosted option that will be forever open-source.

Self-hosted vs Hosted/Cloud

The core difference at the present between self-hosted and cloud is level of support. We will of course fix bugs and respond to issues on the repo, but hosted users can expect faster responses and quicker resolutions to their issues. Self-hosting also opens the door for unique deployment/infra issues we aren't equipped to help with.

Where is data stored?

The vast majority of data involved in Workflows (especially the configuration forms you see in Workflow Builder) are all stored on Slack's servers, Workflow Buddy just acts as an intermediary when they are running. The data Buddy stores is limited to what is needed for functionality (encrypted OAuth tokens, Slack team/user IDs, etc.), or is useful for your own team's stats (such as usage of Step types). If you are curious to know more detail, check out the "buddy/db.py" file in the open-source repo, which shows all the database models.
